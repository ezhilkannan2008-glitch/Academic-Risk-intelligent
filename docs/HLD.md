# High-Level Design (HLD)

## Architecture Overview
The system is built using a decoupled architecture with a React frontend and a Flask backend, communicating via RESTful APIs.

1. **Frontend (React)**: Handles data presentation, routing, and user interactions. Uses a glassmorphism design system to present risk metrics clearly and beautifully.
2. **Backend (Flask)**: Exposes endpoints for retrieving course information and computed risk metrics.
3. **Analytics Engine (Python)**: Processes raw data to compute statistical measures (average, variance, clustering).
4. **Database (SQLite)**: Stores students, courses, scores, and submission records.

## Data Flow
1. **Client Request**: The frontend makes an HTTP GET request to `/risk-analysis` (for Dashboard) or `/course/<id>` (for Course Detail).
2. **Controller (Flask Route)**: Receives the request and calls the appropriate service.
3. **Analytics Service**:
   - Fetches raw scores and submissions from the SQLite database.
   - Computes statistical metrics (`avg_score`, `std_dev`).
   - Identifies anomalies (e.g., flags "Easy Exam" if avg > 85).
4. **Risk Service**:
   - Aggregates the flags generated by the analytics service.
   - Calculates a weighted `risk_score` (0-100).
   - Generates human-readable explanations based on the flags.
5. **Response**: The aggregated data is returned to the frontend as JSON.
6. **Visualization**: The frontend parses the JSON and renders it using Recharts and custom CSS components.

## Database Design
- `students` (`id`, `name`)
- `courses` (`id`, `course_name`)
- `scores` (`student_id`, `course_id`, `score`)
- `submissions` (`id`, `student_id`, `course_id`, `timestamp`)

*Note: In the data generation, we treat `course_id` similarly to an `assignment_id` to keep the model focused on course-level analysis.*
